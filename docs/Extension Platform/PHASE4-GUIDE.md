# Phase 4 — Completing the contract: the implementation guide

**Follow this document top to bottom.** It assumes you have never seen this
project. Everything you need is here or linked from here.

## 0. Orientation — 5 minutes

**Grain** is a local, low-RAM speech-to-text desktop app. Rust/Tauri backend in
`src-tauri/`, React frontend in `src/`, shared Rust crates in `crates/`, and a
separate native Rust window (the "pill") in `crates/grain-pill/`. It is built on
an upstream project called *Handy*, which lives verbatim in
`src-tauri/src/handy/` — **never add features there**, only `[GRAIN]`-marked
hooks.

**Extensions** add features without forking Grain. Three tiers — `pack` (data
only), `scripted` (JavaScript in a Web Worker), `native` (a separate process).
Every power is a **capability** declared in a manifest and enforced **in Rust**
on every message over a local WebSocket. Extensions never create windows; they
declare *surfaces* and the host builds, sleeps and destroys them.

**What already works (phases 0–3.5, all shipped):** the security wall, the
scripted runtime with per-extension workers and tokens, schema settings,
host-owned windows and overlays, pill themes, and a full developer mode —
`grain-ext` CLI (`init`/`dev`/`doctor`/`pack`), load-unpacked, sub-second hot
reload, source-mapped errors, typed host errors, and a filtered developer log
console.

**Today's capability vocabulary** (`KNOWN_CAPABILITIES` in
`crates/grain-sdk/src/manifest.rs`) is deliberately narrow:

```
events:sessions  events:transcripts  transform:transcript  session:start*
storage  settings  llm  embed  capture:selection
surface:workspace  surface:overlay  pill:slots
```

`session:start` is marked `*` because it is **reserved but not implemented** —
`host_api.rs` capability-checks it and then returns a typed "not implemented".
That is the hole this phase fills first.

### What Phase 4 is for, in one sentence

Phase 3 gave extensions a **face**; Phase 4 gives them **agency** — the ability
to own a recording session, to talk to the outside world, and, when the contract
genuinely cannot help, to ship their own program.

### Repo facts that would otherwise cost you an hour

| Thing | Fact |
|---|---|
| Build the backend | Run `cargo` **from inside `src-tauri/`** — its `.cargo/config.toml` sets a short `target-dir`. A long path makes whisper.cpp's CMake fail on Windows MAX_PATH. |
| Never do | Launch a second app instance while the user's Grain is running — port 7124, global shortcuts and the target dir all collide. |
| Tests | `cargo test` in `src-tauri`, plus `crates/grain-sdk`, `crates/grain-pill`, `crates/grain-core`. Frontend: `npx tsc --noEmit`. All green at the start of this phase. |
| `src/lib/bindings.ts` | **Generated by specta — never hand-edit.** New commands: raw `invoke()` with a local mirror type until a dev run regenerates it. |
| `src/extension-runtime.ts` | The worker shim is a **backtick template literal**. A backtick anywhere inside it — even in a comment — silently terminates the string. |
| Upstream ratchet | Only if you touch `src-tauri/src/handy/`: commit first, then `python Upstream/ratchet.py`; on justified growth `--update` + a follow-up commit. |
| Git | Commit each verified chunk promptly. **Never rebase** — on a rejected push use `git pull --no-rebase`. Clean human commit messages, no AI attribution. |
| Tauri capabilities | A window with no matching file in `src-tauri/capabilities/` **cannot call `listen()`**; app commands are allowed everywhere by default. This asymmetry has bitten this project once already. |
| Secrets | `grain-core` already keeps **secrets in a separate credential file**, apart from settings JSON. Reuse it (step 4) — do **not** add an OS-keychain dependency. |

## 1. What is in this phase, and what is deliberately not

This is the most important section in the guide. Phase 4 is where a platform
either grows the powers it needs or bloats with powers nobody asked for.
[CAPABILITY-GOVERNANCE.md](CAPABILITY-GOVERNANCE.md) draws the line:
**structural** capabilities change the *shape* of what an extension can be, and
must land early or never; **additive** ones are one more thing to touch, so we
reserve the name and design the shape when a real consumer exists — because you
cannot test a guessed interface.

### In scope

| # | Step | Why it earns a place |
|---|---|---|
| 1–2 | **`session:start` + `contributes.sessionMode`**, and making session ownership visible | The last **structural** gap. Lands now or never |
| 3 | **`net:<host>`** — host-proxied, per-host consent | Without it a scripted extension can talk to nothing, and authors escalate to native tier for the sake of one HTTP call |
| 4 | **Extension secrets** in the existing credential file | `net:` without a safe place for an API key is a footgun (correction **C-8**) |
| 5 | **Per-worker memory ceiling** | Grain's whole premise is low RAM; today only *latency* is bounded (correction **C-7**) |
| 6 | **Tier-C `companion`** — the extension ships its own program | The freedom-ladder promise: "if the contract can't do it, build it yourself" |
| 7 | **Acceptance** — the Grain Space Test's session row, proven by a real extension | The only evidence that any of this works |

### Deliberately NOT in scope

Each of these is *named and reserved* — an author hits a documented "not yet",
never an unexplained absence. **Do not build them because they are listed in an
old plan. Build them when someone needs them.**

| Deferred | Why | What triggers it |
|---|---|---|
| **`provides:` / the provider broker** | Extension-to-extension interface sharing, with zero consumers. Real machinery for a hypothetical | A second extension wants what a first one implements |
| **`surface:settings-panel` (custom settings iframe)** | The declarative schema has covered every case so far. Phase 3's realm plumbing makes it ~a day whenever it is needed | An extension's settings genuinely cannot be expressed declaratively |
| **`screen:capture`, `surface:pointer`, `audio:play`** | **Step 6 is the answer to these.** A companion process can screenshot, draw and play audio using the OS APIs directly — that is precisely what rung 4 of the freedom ladder is for | Several extensions want the *same* one, which is the signal it should be a shared host capability |
| **`open:url` / `open:app`, `clipboard:*`** | Danger-marked, no consumer. `capture:selection` already covers the read-side case that exists | A concrete extension needs one |
| **`contributes.promptLayer`, pill **action chips**, `overrides:*`, `resident`** | Additive, no consumer. The pill's *theme* half shipped in Phase 3; chips are a different API | A concrete extension needs one |
| **Re-platforming built-in features onto the contract** | A refactor whose only beneficiary is our own confidence, with real regression risk to features users rely on today | A built-in needs to change anyway, or an author asks for a primitive only the refactor would expose |

> **Why deferring `screen:capture` is safe and not a cop-out.** Step 6 ships the
> companion tier. An author who needs screenshots writes a small program that
> takes them — the OS polices that program, exactly as it polices any app. What
> Grain controls is the door between that program and Grain: it still only
> receives the transcripts and sessions the user approved. That is the design's
> whole answer to "what if Grain can't do the thing I need", and it means we
> never have to guess at a screen-capture interface before anyone has used one.

### One decision to make before you start

**Steps 1–5 are the phase. Step 6 (companion tier) is the largest single item in
it and has no confirmed first consumer.** If time is short, ship 1–5, and let
step 6 wait for the first author who actually hits the wall. Nothing in 5A or 5B
depends on it. If you do build it, remember the hard boundary in §3 below.

## 2. Steps

### Step 1 — `session:start` and `contributes.sessionMode`

**Why.** Today an extension can *react* to a recording (transform the text
afterwards, within a 150 ms budget). It cannot *own* one. That single gap is why
the Grain Space Test has an open row, why a voice-note extension cannot exist,
and why the slow-stage story in [CASE-HEYCLICKY.md](CASE-HEYCLICKY.md) is
unbuildable. It is structural: every later capability is shaped by whether an
extension can own a session, so it lands now.

**The shape.** The host keeps owning the microphone. An extension declares a
**session mode**; the host starts a session in that mode; while it runs, the
extension owns the **slow stage** — the seconds-long stage that core
post-processing occupies today.

```jsonc
"contributes": {
  "sessionMode": { "id": "note", "label": "Dictate a note",
                   "default_binding": "Ctrl+Shift+N" }
}
```

Two ways in, one code path: the declared shortcut fires, **or** the extension
calls `session.start({ mode: "note" })` with the `session:start` grant.

**Where.** `crates/grain-sdk/src/manifest.rs` (the declaration, and add
`sessionMode` to `Contributes`) · `src-tauri/src/host_api.rs:633` (replace the
`not_implemented("session.start")` arm — the preflight and capability mapping
already exist) · `src-tauri/src/extension_host.rs` (dispatch the stage to the
owning worker) · `src-tauri/src/grain_post_process.rs` (the slow-stage hook —
this is Grain code, not the Handy tree).

**What to build.**

1. **Ownership, serialized.** Exactly one session exists at a time — SPEC §3.2
   makes the recording session a hard singleton. `session.start` while one is
   running fails with a typed `E_SESSION_BUSY`. It never queues silently, and it
   never interrupts a user's dictation.
2. **The slow stage.** When an extension owns the session, its stage handler
   receives the transcript with a **long** budget (seconds, not the transform
   pipeline's 150 ms) and returns either replacement text or "handled — suppress
   output". Bound it anyway: a hard ceiling around 30 s, then fail cleanly and
   fall back to normal output. A hung extension must never eat a user's words.
3. **Cancellation is not optional.** If the user cancels, or the extension is
   disabled or reloaded mid-stage, the worker gets a cancel signal and the host
   stops waiting immediately. Test the reload case specifically — hot reload is
   shipped, so it *will* happen mid-session during development.
4. **Failure returns the user's text.** Any error, timeout or crash in the stage
   falls back to Grain's normal output path with the untouched transcript. The
   guarantee to state in the docs: **owning the stage can change your words; it
   can never lose them.**

**Done means.** A dev extension declares a mode, binds a shortcut, receives the
transcript in its stage, returns modified text, and that text is pasted. A second
`session.start` during it returns `E_SESSION_BUSY`. Killing the worker mid-stage
still pastes the original transcript. Each of those four is a test.

### Step 2 — Make session ownership visible

**Why.** Step 1 lets a third party start a recording and see everything said
into it. **A user must always be able to tell that the session running is not
Grain's, and whose it is.** This is not a nicety attached to step 1 — it is the
condition that makes step 1 acceptable to ship.

**Where.** The pill (`crates/grain-pill/`) already renders session state and is
already a WebSocket client receiving `DaemonEvent`s.

**What to build.** Extend the session-start event with the owning extension's
display name, and have the pill show a plain indication that a named extension
owns this session. Grain's own sessions look exactly as they do today.

**Keep it host-rendered and data-only.** This is *not* the contributed pill-chip
API (deferred in §1) — no extension code, no extension-supplied layout, no new
capability. The host states a fact it already knows. That is a few lines; the
chip API is a subsystem.

**Done means.** Starting a session from an extension shows its name; starting
one normally is visually unchanged. The extension cannot influence the indicator
beyond its own name, which the user already approved at install.

### Step 3 — `net:<host>`

**Why.** A scripted extension today can reach exactly one outside service, the
LLM router, through `llm`. Anything else — a weather API, a note service, a
webhook — is impossible. The practical consequence is worse than the missing
feature: **authors escalate to the native tier to make one HTTP call**, which
trades a sandboxed worker for an unsandboxed process. Shipping `net:` keeps
people *out* of native tier, so it is the higher-leverage capability even though
native sounds more powerful.

Unlike the deferred items, this has a settled shape — SPEC §1.3 already
specifies it as "proxied fetch, per host, host shown in consent". There is
nothing to guess.

**Where.** `crates/grain-sdk/src/manifest.rs` (`KNOWN_CAPABILITIES` must accept
the parameterised `net:<host>` form) · `src-tauri/src/host_api.rs` (the proxy) ·
the permission sheet in the Extensions UI.

**What to build.**

1. **The grant names the host, and the sheet shows it.** `net:api.example.com`
   reads as "can send data to api.example.com" in the consent sheet. A wildcard
   host is rejected at import — an extension that wants to talk to anything is
   telling you something.
2. **The fetch happens in Rust, never in the worker.** The worker gets no socket;
   it asks the host, which validates the URL's host against the grant *after*
   parsing (never by string prefix — that is how `evil.com/api.example.com`
   sneaks through), applies a timeout and a response-size cap, and returns the
   result.
3. **Redirects are re-validated**, every hop, against the same grant. A redirect
   to an ungranted host fails rather than following.
4. **No credential passthrough.** The host never attaches Grain's own API keys,
   cookies, or auth headers to an extension's request.

**Done means.** Tests cover: a granted host succeeds; an ungranted host fails
typed; a redirect off-grant fails; a suffix-confusion URL fails; oversized
responses are truncated or refused. The consent sheet names the host in plain
words.

### Step 4 — Extension secrets (correction C-8)

**Why.** Step 3 makes API keys inevitable, and there is currently nowhere safe to
put one — settings values live in plain JSON.

**Where.** `crates/grain-core` **already keeps secrets in a separate credential
file**, apart from the settings store. Reuse that. Do not add an OS-keychain
dependency for this; it is a new dependency and a new failure mode for a problem
this project already solved.

**What to build.** A `secret` setting kind in the schema. Values route to the
credential store under `ext.<id>.<key>`, never into `extensions.json` or the
settings JSON. The UI renders a masked field. **A secret is write-only to the
extension**: it can set one and ask the host to *use* one, but reading it back
returns a redacted marker. Purge on uninstall, with the rest of the extension's
data.

**Done means.** A secret set through the UI never appears in any settings file
on disk; a test asserts that. Uninstall removes it.

### Step 5 — Per-worker memory ceiling (correction C-7)

**Why.** The transform pipeline bounds how *long* an extension may take, and
nothing bounds how much memory it may hold. On the edge devices this app targets,
one leaking extension is a Grain problem regardless of whose bug it is. Raycast
enforces a per-extension memory limit and stops over-budget extensions; we bound
latency and should bound the other axis too.

**Where.** `src-tauri/src/extension_host.rs` (the reaper and strike machinery are
already there) and the JS supervisor.

**What to build.** Sample each worker's reported heap on the existing reaper
tick — WebView2 is Chromium, so a heap reading is available — and apply the
**same strike semantics as the transform timeout**: over budget repeatedly →
disable with a clear, typed reason the user can read, and a `life` line in the
developer log console.

**Be honest about the limitation** in a comment: this reads what the JS engine
reports, which is not the process's true footprint. It catches a runaway
allocation, which is the failure that matters; it is not an accounting system.
Set the ceiling generously — this exists to stop pathology, not to police
authors.

**Done means.** A deliberately leaking test extension is disabled with a legible
reason, and a well-behaved one is never touched by this code path.

### Step 6 — Tier-C `companion` (the largest item; see §1's note)

**Why.** This is the platform's answer to "what if Grain can't do what I need" —
and having a real answer is what lets us decline speculative capabilities without
that being a dead end. The author ships a small program; **inside it, the OS is
the sandbox, exactly as for any app**; at Grain's boundary, everything is still
capability-checked.

**Build `companion` only. Do not build `provider`/`provides:`** — that is
extension-to-extension interface sharing with no consumer (§1).

**Where.** A new supervisor module in `src-tauri/`, modelled on how the pill
process is already spawned and supervised. Manifest: `tier: "native"` with a
per-platform binary map.

**What to build.**

1. **Same door, different runtime.** The companion connects to the same local
   WebSocket and authenticates with its own minted token. **No new protocol, no
   new trust path, no Tauri IPC** — everything already true of a worker stays
   true here. Identity remains bound to the channel.
2. **Supervised like the pill.** Spawn on activation, kill on disable, uninstall
   or app exit. Crash → restart with backoff, then give up and surface it. **No
   orphan survives Grain exiting** — verify by killing Grain uncleanly and
   checking the process table. This is the failure everyone ships at least once.
3. **The binary is data until it is verified.** In Phase 4 a companion runs
   **only under developer mode**, from a folder a human chose. It is not
   installable from anywhere else, because the trust rails that would make that
   safe are Phase 5A. Enforce this in code, not in documentation.
4. **The card says so.** Any extension with a native component is labelled as
   such, before install and in the list.

**Done means.** A companion starts on activation, exchanges capability-checked
messages, dies with its extension, and leaves nothing behind after an unclean
Grain exit. A companion in a non-dev install path is refused.

### Step 7 — Acceptance: close the Grain Space Test's session row

**Why.** [PHASE3-REVIEW.md](PHASE3-REVIEW.md) walked the platform's acceptance
bar — *could a third party build Grain Space?* — and found 8 of 10 rows
satisfied, with `session:start` the one structural gap. This step closes it with
evidence rather than assertion.

**What to do.** Using the Phase 3.5 developer loop, build a **voice-note
extension**: a shortcut starts a session in its mode, the transcript arrives in
its slow stage, it structures the text with `llm`, stores it with `storage`, and
suppresses the paste because it handled the output itself. Then update
PHASE3-REVIEW.md's table — honestly. If a row still does not pass, record why.

**Done means.** The session row is satisfied; the review doc reflects reality;
and the extension lives in the examples directory so the next author has a
working model to copy.

## 3. The boundary Phase 4 must not cross

**A native extension is loadable in developer mode and is not distributable.**
The trust rails — signed index, pinned key, verified artifacts, revocation — are
Phase 5A. Shipping a way to distribute programs before the machinery that proves
where they came from is exactly the wrong order, and it is the mistake this
project's whole distribution design exists to avoid.

Everything else in Phase 4 is distributable the moment 5A and 5B land.
